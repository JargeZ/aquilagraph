# Исследование: позиционирование субграфов в Graphviz

## Текущее состояние проекта

В [`graph-builder.ts`](src/core/graph/graph-builder.ts) граф строится так:
- `rankdir = "LR"` (ранги слева направо)
- Каждый модуль -- `cluster_<name>` (субграф с `style=dashed`)
- Ребра между модулями -- обычные edge на верхнем уровне digraph
- Движок: `dot` (задаётся в [`dot-svg-canvas.tsx`](src/components/dot-svg-canvas.tsx))
- **Не используются**: `compound`, `lhead`/`ltail`, `newrank`, `ratio`, `pack`, `splines`, `constraint`

## Что говорит документация Graphviz о позиционировании кластеров

### 1. Как dot располагает кластеры (алгоритм)

Dot использует **рекурсивный** подход (`clusterrank=local`, по умолчанию):
1. Каждый кластер раскладывается **отдельно** (как маленький граф внутри себя)
2. Затем кластер интегрируется **как единое целое** (bounding rectangle) в родительский граф
3. Позиция кластера определяется рангами его узлов -- кластер занимает те ранги, которые заняли его внутренние узлы

**Ключевая проблема**: dot не рассматривает кластеры как "большие ноды" при взаимном позиционировании. Кластер -- это просто прямоугольник вокруг своих узлов. Если нет рёбер между конкретными узлами двух кластеров, dot не имеет причин размещать их рядом в определённом направлении. При `rankdir=LR` кластеры без межкластерных рёбер складываются вертикально -- потому что они все попадают на одни и те же ранги.

### 2. Атрибуты, влияющие на позиционирование кластеров

#### `compound=true` + `lhead`/`ltail` (dot)
- Позволяет рёбрам "приходить в кластер" как в единицу (ребро обрезается на границе кластера)
- **Влияние на layout**: рёбра всё равно привязаны к конкретным узлам внутри кластеров (head/tail), поэтому на ранжирование это влияет слабо -- оно уже определяется обычными рёбрами между узлами
- **Визуальный эффект**: стрелки заканчиваются на границе кластера, а не на ноде -- выглядит как "связь между модулями"

#### `newrank=true` (dot)
- Использует единый глобальный ранг, игнорируя границы кластеров
- Может дать больше рангов и более широкую раскладку, но может также сломать внутреннюю структуру кластеров (head выше tail)
- Не решает основную проблему "шпиля"

#### `ratio` (dot)
- `ratio=<число>` -- масштабирует drawing в одном измерении чтобы достичь нужного отношения сторон. Например `ratio=1` делает квадрат, но это **масштабирование**, а не перераскладка
- `ratio="compress"` + `size` -- пытается сжать layout, жертвуя балансом и симметрией. Работает только с dot
- `ratio="fill"` + `size` -- масштабирует координаты узлов, чтобы заполнить заданный размер
- **Ограничение**: ratio и size масштабируют/искажают готовый layout, а не меняют взаимное расположение кластеров

#### `rank=same|min|max|source|sink` (dot)
- Заставляет узлы субграфа быть на одном ранге. Уже используется для бакетов классификаций
- Не помогает напрямую для позиционирования кластеров-модулей друг относительно друга

#### `ordering` (dot)
- Управляет порядком рёбер (left-to-right), но не позицией кластеров

### 3. Альтернативный движок: `osage`

**Osage -- единственный движок, который делает именно то, что описано в задаче**: "субграфы ведут себя как ноды".

Цитата из документации:
> osage draws the graph recursively. At each level, there will be a collection of nodes and a collection of cluster subgraphs. The internals of each cluster subgraph are laid out, then the cluster subgraphs and nodes at the current level are **positioned relative to each other, treating each cluster subgraph as a node**.

- Кластеры и узлы рассматриваются как прямоугольники для упаковки (packing)
- Управляется атрибутами `pack` и `packmode` (на каждом уровне)
- Рёбра маршрутизируются **после** позиционирования (через `splines`)
- Поддерживается в `@viz-js/viz` (вместе с dot, fdp, neato, circo, twopi)

**Ограничения osage**:
- Рёбра **игнорируются** при packing -- т.е. взаимное положение кластеров **не отражает связи** между модулями, только минимизирует пустое пространство
- Нет иерархического ранжирования -- пропадает семантика "source -> sink"
- Маршрутизация рёбер хуже, чем в dot (могут быть пересечения)

### 4. Другие движки

- **fdp / neato / sfdp** -- force-directed layout. Не поддерживают кластеры в полноценном виде. fdp имеет ограниченную поддержку кластеров, но без гарантий
- **twopi** -- радиальный layout, не подходит
- **circo** -- круговой, не подходит

### 5. Утилита `unflatten`

Препроцессор, улучшающий aspect ratio путём добавления invisible edges с разным `minlen`. Работает **до** dot. Помогает для графов с множеством "листьев", но не решает проблему расположения кластеров.

## Выводы: что реально может помочь

### Вариант A: Гибридный подход (dot + невидимые рёбра)
- Добавить `compound=true` и invisible edges между кластерами (через `lhead`/`ltail`) чтобы повлиять на ранжирование
- Проблема: нужно вычислять "межмодульные" связи и создавать рёбра уровня кластера, а не узла

### Вариант B: Переключение на `osage` для верхнего уровня
- Квадратная упаковка "из коробки"
- Потеря информации о направлении связей между модулями
- Внутри кластеров можно задавать `layout=dot` -- osage это поддерживает

### Вариант C: Двухпроходный layout
- Сначала построить "мета-граф" модулей (модуль = нода, ребро = есть связь между модулями)
- Разложить мета-граф через dot/fdp -> получить координаты модулей
- Использовать эти координаты как подсказки для основного графа (через `pos` + `-n` flag)
- Самый сложный, но самый точный

### Вариант D: `ratio="compress"` + `size` (quick fix)
- Не меняет логику, просто сжимает в заданную область
- Может ухудшить читаемость крупных графов
