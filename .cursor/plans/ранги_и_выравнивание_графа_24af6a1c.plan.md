# Ранги графа по типам и варианты DOT/Graphviz

## Текущее состояние

- DOT строится в `[src/core/graph/graph-builder.ts](src/core/graph/graph-builder.ts)`: `rankdir = LR`, узлы в кластерах `cluster_*`, рёбра — по `uses`. **Явных ограничений ранга по типу нет** — порядок колонок задаётся в основном направлением рёбер и эвристикой `dot`.
- Вкладка React Flow получает ту же модель через `digraphToFlow` и затем **перекладывает Dagre**; после layout вызывается `[repositionByType](src/core/graph/digraph-to-flow.ts)` — сейчас там частично тянутся **controlling** влево внутри модуля и выстраивается ряд **sideEffect**, без явного слоя для **unclassified** и **businessLogic**.

Желаемый приоритет колонок (слева → направо):

1. `controlling` (Controlling)
2. `unclassified`
3. `businessLogic`
4. `sideEffect` (SideEffect)

## Как в DOT/`dot` задаётся «выравнивание» (на что опираться при выборе)

Движок `**dot`** — иерархический: при `rankdir=LR` «ранги» — это **вертикальные полосы (колонки)** слева направо; рёбра в основном тянут хвост левее головы. Ниже — рабочие приёмы из [атрибутов Graphviz](https://graphviz.org/doc/info/attrs.html) и практики.

### A. Невидимый «хребёт» + `rank=same` с якорями (жёсткие 4 колонки)

- Идея: завести 4 **невидимых** узла-якоря `L0→L1→L2→L3` с рёбрами `style=invis` (и при необходимости `weight` высоким), затем для каждого типа: подграф `{ rank=same; Lk; все узлы типа k }`.
- **Плюсы:** наиболее прямой способ получить **ровно четыре колонки** в нужном порядке независимо от модулей.
- **Минусы:** при **циклах** или рёбрах «справа налево» относительно желаемого порядка (`sideEffect` → `controlling` и т.д.) `dot` может не найти допустимое ранжирование или сильно «разъедет» граф; иногда помогают `constraint=false` на отдельных рёбрах (см. вариант D).

### B. Только края: `rank=min` и `rank=max` на подграфах

- `rank=min` — узлы подграфа к **минимальному** рангу (в `LR` это **левый край**), `rank=max` — к **правому**.
- **Плюсы:** мало кода, не засоряет граф якорями.
- **Минусы:** гарантируется в основном **«все controlling слева»** и **«все sideEffect справа»**; порядок **двух средних** слоёв (`unclassified` vs `businessLogic`) **не фиксируется** без дополнительных рёбер/якорей.

### C. Невидимые рёбра «тип_i → тип_{i+1}» (выборочно или полный мат)

- Идея: добавить невидимые рёбра от представителей левого типа к правому, чтобы потянуть ранги. Можно **одно ребро на пару типов** между «стоком» левого слоя и «истоком» правого — экономнее полного декартова произведения.
- **Плюсы:** гибко усиливать порядок без явных `rank=same` для каждого узла.
- **Минусы:** нужно аккуратно выбирать пары узлов (иначе лишние перекрёстные рёбра портят компактность); при конфликтах с реальными рёбрами снова нужны `constraint=false` / `weight`.

### D. Смягчение конфликтов: `constraint=false`, `weight`, `minlen`

- `**constraint=false`** на ребре — ребро рисуется, но **не участвует** в ранжировании (полезно для обратных зависимостей, которые ломают колонки).
- `**weight`** — насколько сильно ребро влияет на ранги (меньший вес = слабее «тянет»).
- `**minlen`** — минимальное число уровней между хвостом и головой (можно использовать для раздвижки слоёв).
- **Плюсы:** спасают раскладку при сложных графах.
- **Минусы:** это уже **настройка под данные**, не «одна декларация порядка».

### E. Порядок узлов **внутри одной колонки** (не слева-направо между типами)

- Граф: `**ordering=out` / `ordering=in`** — влияет на порядок следования узлов **на одном ранге** (в `LR` это порядок по **вертикали** внутри колонки).
- Узел: атрибут `**group`** — узлы с одинаковым `group` стараются оказаться **на одном ранге** (удобно для «держать вместе», не для 4 колонок по типу).
- `**ranksep` / `nodesep`** — в основном **расстояния**, не семантика колонок.

### F. Другие движки (`neato`, `fdp`, `sfdp`)

- Дают **силовую/геометрическую** укладку; **колонки по типу** там не задаются декларативно как в `dot`. В проекте рендер идёт через `engine: "dot"` в `[dot-svg-canvas.tsx](src/components/dot-svg-canvas.tsx)`, смена движка — отдельное продуктовое решение.

## Рекомендация по выбору (кратко)


| Цель                                | Разумный выбор                                          |
| ----------------------------------- | ------------------------------------------------------- |
| Строго 4 колонки в заданном порядке | **A** (+ при необходимости **D**)                       |
| Только «края» проще и дешевле       | **B** + точечно **C** для средних слоёв                 |
| Минимум искусственных узлов         | **C** с одним-двумя невидимыми рёбрами на границу слоёв |


Практически для вашего полного порядка из 4 типов чаще всего берут **A** или **B+C** для середины.

## Реализация в коде (после выбора варианта)

- **Точка изменений DOT:** `[buildGraph` в `graph-builder.ts](src/core/graph/graph-builder.ts)` — после объявления узлов и рёбер `uses` добавить невидимые узлы/подграфы с `rank` или невидимые рёбра (в `ts-graphviz` атрибуты подграфа задаются через `sg.set(...)`; при отсутствии ключа в типизированном `attribute` допустима строка вида `'rank'` / значение `'same'`, как в сыром DOT).
- **Идентификаторы якорей:** префикс вроде `__lane0__` … `__lane3__`, чтобы не пересекаться с `reference` элементов; в DOT при необходимости — экранирование/кавычки.
- **Пустые типы:** якоря цепочки оставить даже если слой пустой, иначе ранги схлопнутся.
- **Тесты:** обновить снапшоты в `[graph-builder.test.ts](src/core/graph/__tests__/graph-builder.test.ts)` и при необходимости `[analyze.test.ts](src/core/__tests__/analyze.test.ts)`.
- **React Flow:** если нужен **тот же** визуальный порядок, что и в SVG, расширить `[repositionByType](src/core/graph/digraph-to-flow.ts)` (или заменить на общую функцию порядка по X по `elementType` внутри модуля/глобально) — иначе изменения только в DOT не затронут Dagre-вкладку.

## Риски

- **Циклы и обратные рёбра** относительно желаемых колонок — основной источник «ломаной» укладки; без **D** может потребоваться ослабить требование или помечать проблемные рёбра.
- Кластеры **не отменяют** глобальные `rank`-ограничения, но плотная вложенность может визуально усложнять чтение — после внедрения имеет смысл посмотреть реальный проект в UI.
